# 開發歷程：product-image-resizer

> 給電商店家的批次商品圖轉檔工具網站——零上傳、零營運成本

---

## 🎯 需求起源（2026-07-07）

電商平台的店家常有一批照片要上架，但系統對商品圖有尺寸與檔案大小要求。店家自己不會（或懶得）用修圖軟體逐張處理，需要一個工具網站：

1. **批量**處理照片
2. **等比例縮放**，不能破壞比例（不變形）
3. 符合系統的**尺寸**與**檔案大小**上限
4. 用**免費流量**跑，而且要能「限制流量」避免太多人用導致站主要付費

## 💡 關鍵決策：純前端處理（Client-side）

**問題**：如果照片上傳到伺服器處理，流量與運算成本跟用量成正比，免費方案很快爆掉，還要做限流。

**評估過的方案**：
| 方案 | 流量成本 | 爆量風險 |
|---|---|---|
| 後端處理（Cloud Run / Lambda） | 上傳+下載+運算全算你的 | 高，需限流與帳單警報 |
| ✅ **瀏覽器內處理（Canvas API）** | **趨近於零**（只載一次靜態頁） | **無**，運算用店家自己的電腦 |

**決定**：純前端。照片從頭到尾不離開使用者的瀏覽器——「限制流量」的需求直接消失，因為根本沒有會產生費用的流量。附帶好處：不用等上傳、隱私安全（商品圖不經第三方）。

**託管選擇**：GitHub Pages（免費 100GB/月頻寬、**超量只限流不收費、沒有帳單機制**）。Cloudflare Pages 也可（頻寬無上限），但用戶已有 GitHub 工作流程，Pages 部署 = git push，最順。

**「限制對象」取代「限制流量」**：如果之後不想讓工具外流，加一組前端通行碼即可（跟店家公告，改版換碼）。首版先不加。

## 🛠️ 實作要點

- **等比縮放**：`scale = min(targetW/w, targetH/h, 1)`——只縮小不放大，比例永遠不變
- **兩種輸出模式**：
  - `pad`（預設）：等比縮放後置中、補底色（預設白），輸出固定尺寸（如 1000×1000）——商品圖最常見的需求
  - `fit`：等比縮至長邊，輸出保持原比例
- **畫質**：大圖一次縮到小尺寸會糊 → **分段減半縮小**（step-down halving）+ `imageSmoothingQuality: "high"`
- **手機直拍轉向**：`createImageBitmap(file, { imageOrientation: "from-image" })` 自動處理 EXIF，失敗退回 `<img>` 載入
- **壓到大小上限**：`canvas.toBlob` 品質從 0.92 逐級降到 0.4，取第一個 ≤ 上限的結果；PNG 沒有品質參數，超標直接提示改用 JPG
- **批次下載**：JSZip 打包，同名檔自動加流水號
- **規格未定的處理**：用戶尚未提供平台的正式圖片規格 → 尺寸/大小上限/格式全部做成頁面可調設定（預設 1000×1000、白底補邊、≤1MB、JPG），規格確定後再鎖成預設值
- `noindex`：不讓搜尋引擎收錄，工具只給自家店家用

## 📋 規格預設值

| 項目 | 預設 | 可選 |
|---|---|---|
| 尺寸 | 1000×1000 | 800 / 1200 / 自訂 |
| 縮放 | 等比＋白底補邊 | 等比縮至長邊 |
| 大小上限 | 1 MB | 300KB / 500KB / 2MB / 不限 |
| 格式 | JPG | WebP / PNG |

## 📐 正式規格定案（2026-07-07）

用戶問到 OrderPally 官方規格：**尺寸不限、檔案 ≤ 2MB**。調整：
- 預設改「維持原尺寸（只壓縮檔案）」+ 上限 2MB；原尺寸模式下隱藏縮放方式/底色選項（無意義）
- 尺寸選項仍保留（800/1000/1200/自訂），店家想順便縮圖也行
- **保險機制**：極大圖（例如單眼原檔）壓到品質下限 0.4 仍超過 2MB 時，自動每輪縮小 20% 尺寸再壓，直到符合（最多 6 輪或短邊 300px 止）——規格尺寸不限所以縮尺寸可接受
- 原尺寸模式輸出檔名後綴 `_compressed`

## ✂️ 極簡化改版（2026-07-07 稍後）：拿掉所有選項

用戶實際體驗後給了更明確的產品方向：**店家不需要選任何東西**。
> 「不要有尺寸選項 也不要有 mb 選擇 直接判斷照片有沒有超過 2MB 有就改 沒有就一起還給使用者」

- 整個「輸出規格」設定區塊移除，流程剩兩步：拖圖 → 下載
- 邏輯：≤2MB **原檔原名保留（位元組都不動）**；>2MB 維持原尺寸壓成 JPG（品質 0.92→0.4 逐級），仍超標每輪縮 20% 尺寸
- 兩種結果混在同一清單、一起打包 ZIP；清單標示「原檔保留」vs「已壓縮 X → Y」
- **修 bug**：舊版 `fmtSize` 未定義（顯示結果時會 JS 錯誤）——重寫時發現，這可能就是用戶回報「網站沒更新/沒反應」的真因
- 加 `?v=3` 快取破壞參數，避免店家瀏覽器抓到舊版 js/css

**教訓**：工具給非技術用戶，預設值再好都不如「沒有選項」。第一版的彈性設定是給自己看的，不是給店家用的。

## 🐛 Bug 修復：大尺寸 PNG 轉檔失敗（2026-07-08）

### 問題回報

用戶回報「Desktop 1441.png 轉檔失敗（瀏覽器不支援）」——檔名是 PNG，錯誤出現在畫面清單。

### 根因分析

錯誤訊息來自 `compressToLimit()` 裡的 `canvas.toBlob()` 回傳 `null`。

追蹤流程：
1. `loadBitmap()` 成功（圖片有正確載入）
2. `makeCanvas(bmp.width, bmp.height)` 建立 canvas
3. `canvas.getContext("2d")` 取得繪圖環境
4. `ctx.drawImage(bmp, 0, 0)` 繪製圖片
5. `canvas.toBlob(resolve, "image/jpeg", quality)` → **回傳 null**

`toBlob` 回傳 null 最常見原因：**圖片解析度超過瀏覽器 canvas 記憶體上限**。Safari 的上限約 16MP（4096×4096），Chrome 也有類似限制。超大圖（掃描器原圖、高解析度商品照）很容易踩到。

原本錯誤訊息是「瀏覽器不支援」，誤導性高——其實是記憶體問題，不是格式問題。

### 修復方式（commit `31eeeb1`）

`compressFile()` 裡加入 canvas 尺寸保護：

```javascript
var MAX_AREA = 16 * 1024 * 1024; // 16MP 安全上限
if (w * h > MAX_AREA) {
  var s = Math.sqrt(MAX_AREA / (w * h));
  w = Math.floor(w * s);
  h = Math.floor(h * s);
}
```

等比縮到 16MP 以內後再壓縮，確保 `toBlob` 不會因記憶體不足而失敗。另加 `getContext("2d")` 回傳 null 的防守判斷，錯誤訊息改為「圖片解析度可能過高」，更貼近實際原因。

### 順帶確認的格式支援

| 格式 | ≤ 2MB | > 2MB 輸出 | 備註 |
|------|-------|-----------|------|
| JPG  | 原檔 ✅ | JPG ✅ | |
| PNG  | 原檔 ✅ | JPG ✅ | 透明區補白底 |
| GIF  | 原檔 ✅ | JPG ⚠️ | 動圖只保留第一幀 |
| BMP  | 原檔 ✅ | JPG ✅ | 現代瀏覽器 |
| WebP | 原檔 ✅ | JPG ✅ | |
| HEIC | 原檔 ✅ | JPG ⚠️ | Safari OK，Chrome 不穩定 |

## 📋 法律聲明加入（2026-07-08）

### 背景

用戶詢問台灣個資法對此類工具的要求，確認法律義務後決定加入聲明。

### 法律分析結論

此工具屬**法律負擔最輕**的一類網站：

- 純前端處理，伺服器端零資料接觸
- 無帳號、無資料庫、無追蹤 cookie
- 台灣《個人資料保護法》（2025年修正版）的告知義務、安全維護義務、DPO 任命等條文**均不適用**
- 唯一第三方：`cdnjs.cloudflare.com` 載入 JSZip，Cloudflare 可能依其政策記錄連線 IP

### 加入內容（commit `28be5ee`）

頁尾新增隱私聲明區塊（白底小卡），涵蓋：
1. 純前端、不蒐集個人資料聲明
2. Cloudflare CDN 第三方說明（附官方隱私政策連結）
3. 轉換結果免責聲明
